Code First, Granular Testing

Posted on June 1, 2017

Yes, TDD and BDD are great techniques, but is there a way to start testing when you’re a beginner developer and you seem to have a hard time getting into TDD or BDD?

If at first TDD or BDD don’t seem to work for your brain you might want to try a different approach. There is nothing inherently wrong with these two testing paradigms. But I feel a different approach might work better for less-seasoned programmers like me.

Code First Granular Testing works like this:

Start with an architecture overview or basic architectural design. You need to know what you’re doing and where you’re going. Using visual tools like a mind map or a process map application can help.

Explore, figure out and write the first “grain”, ie the first bit of functionality.

Once you’ve figured out how to get it working and you’re relatively satisfied with the result (after several rewrites if necessary) freeze the first grain with the first series of tests.

Next do the same for the second grain but this time do run the tests at each iteration ensuring that grain 2 doesn’t break grain 1.

The cycle continues for grain 3.

After the app is done testing for the next app works this way:

If you’re writing a similar app that reuses parts of the framework, and codes practices that you master already then you can swithc to a Test First Development by reusing and adapting the tests from the prototype.

If the app is different and requires you learn while coding then by all means use Granular Testing: code a grain, freeze it with test, code the next grain
